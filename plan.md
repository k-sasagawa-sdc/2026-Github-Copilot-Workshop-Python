# ポモドーロタイマー実装プラン

## 目的

このドキュメントは、architecture.md と features.md を前提として、ポモドーロタイマーアプリケーションを段階的に実装するための計画をまとめたものである。

実装方針は t_wada の TDD に従い、以下を重視する。

- 失敗するテストから始める
- 最小限の実装でテストを通す
- リファクタリングで設計を改善する
- 常に動作する状態を保ちながら小さく進める

## TDD の基本方針

### 開発サイクル

各機能は次のサイクルで実装する。

1. Red
   - 先に失敗するテストを書く
   - 何を満たせば完了かをテストで明確にする
2. Green
   - テストを通すための最小限の実装を行う
   - この段階では過度に一般化しない
3. Refactor
   - 重複除去と責務分離を行う
   - テストが通ったまま設計を改善する

### 実装単位

- 1 回の TDD サイクルで扱うのは 1 つの振る舞いに限定する
- 30 分から半日で確認できるサイズに分割する
- 1 機能をまとめて実装せず、操作単位、状態単位、表示単位で切る

### テスト観点

今回のアプリでは次の順でテストを厚くする。

1. ドメインロジックのテスト
   - 状態遷移
   - タイマー残り時間計算
   - 長休憩判定
2. アプリケーション層のテスト
   - ユースケース単位の振る舞い
   - 設定反映
   - セッション完了処理
3. API テスト
   - Flask ルート
   - 設定保存 API
   - セッション保存 API
4. UI テスト
   - 最小限の DOM 操作確認
   - 画面統合レベルの確認

## 全体方針

### なぜこの順序にするか

- タイマー精度と状態遷移は後から直しにくいため先に固定する
- UI を先に作り込みすぎると仕様変更に弱くなるため、最初は最小限に留める
- 保存機能はコアロジックが固まってから追加した方が安全である
- 常に動くアプリを維持することで、各フェーズの完了が明確になる

### 各フェーズの完了条件

各フェーズには次の条件を設ける。

- 主要な振る舞いがテストで表現されている
- 実装はそのテストを通過している
- リファクタリング後もテストが維持されている
- 手動確認する観点が明文化されている

## フェーズ横断のテスト戦略

### 優先するテストの種類

- ドメインロジックは単体テストを中心に守る
- Flask のルートと API は統合テストで守る
- UI はすべてを自動化しようとせず、壊れやすい振る舞いに絞ってテストする

### テストで固定すべき重要仕様

- 残り時間の計算方法
- 状態遷移の正しさ
- 長休憩へ切り替わる条件
- 設定値のバリデーション
- セッション保存内容
- 集計結果

### 手動確認を残す領域

- UI モックとの見た目比較
- 通知音の体験
- モバイルでの操作感
- アニメーションや配色の印象

## 実装時の注意点

- 仕様が曖昧なまま大きく作り始めない
- テストしづらいコードを書いたら、設計の歪みを疑う
- 未来の拡張を想定しすぎて、最初から複雑にしない
- UI の都合でコアロジックを複雑化させない
- 1 フェーズごとに動くものを残す

## 最初に着手する具体的な順番

1. [完了] Flask のトップ画面テストを書く
2. [完了] 最小の画面を返す実装を入れる
3. [完了] タイマー表示要素の存在テストを書く
4. [完了] UI シェルを作る
5. [次] 残り時間計算関数のテストを書く
6. フォーカスタイマーの開始処理を作る
7. pause と resume のテストを追加する
8. 状態遷移モジュールへリファクタリングする

この順番なら、最初期から常に動くアプリを維持しつつ、最も重要なタイマーコアへ早く到達できる。
