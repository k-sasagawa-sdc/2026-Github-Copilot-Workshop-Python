# Webアプリケーションアーキテクチャ案

## 目的

このプロジェクトでは、Flask をバックエンド、HTML/CSS/JavaScript をフロントエンドとして利用し、ポモドーロタイマーの Web アプリを構築する。

優先事項は次のとおり。

- まずは短期間で動く MVP を作る
- UI モックに沿って、操作しやすく視認性の高いタイマー体験を実現する
- 将来的に設定保存、統計表示、複数端末同期などを追加しやすい構成にする

## 採用方針

### 全体方針

- アプリは Flask によるサーバレンダリングをベースに構築する
- タイマーのカウントダウンや画面更新はブラウザ側の JavaScript で処理する
- サーバ側は画面配信、設定保存、セッション記録、統計提供を担う
- フロントエンドは TypeScript ではなく JavaScript を採用する

### この構成を採用する理由

- タイマー表示やアニメーションはブラウザ側で動かした方が応答性が高い
- Flask 側に状態保存や集計処理を寄せると、将来の拡張がしやすい
- JavaScript を使うことでビルド導入なしでも実装を開始できる
- 小規模アプリとしては十分に保守しやすく、後から構造化もしやすい

## 想定ディレクトリ構成

ポモドーロ関連のファイルは、プロジェクトルールに従って 1.pomodoro 配下に配置する。

```text
1.pomodoro/
  app.py
  templates/
    index.html
    base.html
  static/

      style.css
    js/
      app.js
```

最初の MVP では、templates と static を中心に始め、Python 側の分割は必要になった段階で進める。

## レイヤー設計

### 1. Presentation 層

対象:

- HTML テンプレート
- CSS
- JavaScript による DOM 操作

責務:

- UI モックに沿ったレイアウト表示
- タイマー表示、進捗表示、ボタン状態の更新
- ユーザー操作の受け付け

### 2. Application 層

対象:

- ポモドーロ開始
- 一時停止
- 再開
- スキップ
- リセット
- セッション完了処理

責務:

- ユースケース単位で処理を整理する
- フロントとバックエンドの橋渡しを行う
- 状態遷移の呼び出し順を統一する

### 3. Domain 層

対象:

- ポモドーロの状態
- 作業時間、休憩時間、長休憩周期
- 状態遷移ルール

責務:

- アプリ固有のルールを 1 か所にまとめる
- 作業から休憩への切り替え条件を管理する
- 不正な状態変更を防ぐ

### 4. Infrastructure 層

対象:

- Flask ルーティング
- SQLite などの永続化
- 設定の読み書き
- セッション記録と統計取得

責務:

- データ保存と取得
- API 提供
- 将来的な同期機能への備え

## 状態管理の設計

このアプリでは、タイマー状態の管理が最重要になるため、有限状態機械として設計する。

### 主な状態

- idle
- focus_running
- short_break_running
- long_break_running
- paused

### 主なイベント

- start
- pause
- resume
- complete
- skip
- reset

### 設計上の原則

- 状態遷移のルールは 1 つのモジュールに集約する
- UI から直接タイマー値を書き換えない
- 残り時間は毎秒減算ではなく、開始時刻と経過時間から再計算する
- ブラウザの非アクティブ化や復帰後も、正しい残り時間を維持する

## フロントエンド設計

JavaScript は責務ごとに分割する。

### app.js

- 初期化処理
- モジュール接続
- イベント登録

### timer.js

- タイマーエンジン
- 状態管理
- 経過時間からの残り時間算出
- tick イベント通知

### ui.js

- DOM 更新
- ボタン状態切り替え
- モード表示の変更
- プログレスリングやカラー切り替え

### storage.js

- ローカル保存
- Flask API との通信
- 設定や統計の取得と保存

### 補足方針

- まずはサーバ保存なしでも動くようにし、必要に応じて API 連携を足す
- 画面表示の即時性を優先し、保存は非同期で行う
- 通知音やブラウザ通知は独立した小さなモジュールとして追加可能にする

## バックエンド設計

Flask は画面配信と API 提供を担当する。

### 想定ルーティング

- GET /
  - メイン画面を返す
- GET /api/settings
  - タイマー設定を返す
- PUT /api/settings
  - タイマー設定を更新する
- POST /api/sessions
  - セッション完了結果を保存する
- GET /api/stats
  - 日次または週次の統計を返す

### バックエンドの役割

- 初期表示に必要な設定を返す
- ユーザー設定を保存する
- 完了したポモドーロの実績を保存する
- 統計データをフロントに返す

### settings テーブル

### sessions テーブル

## 実装フェーズ

### Phase 1: MVP

- 単一画面を表示する
- フォーカスタイマーを開始、一時停止、リセットできる
- JavaScript でタイマー表示を更新する
- 必要最低限のスタイルを適用する

### Phase 2: ポモドーロ完全対応

- 短休憩と長休憩を追加する
- セッションの自動切り替えを実装する
- 周期管理を追加する
- 完了音や通知を追加する

### Phase 3: 永続化

- 設定保存 API を追加する
- セッション記録 API を追加する
- SQLite に保存する

### Phase 4: 統計と改善

- 今日の完了回数や集中時間を表示する
- レスポンシブ対応を調整する
- アクセシビリティを改善する
- UI モックとの差分を詰める

## 品質上の注意点

### タイマー精度

- setInterval の単純減算だけに依存しない
- タイマー復帰時に時刻差分から再計算する

### 状態遷移の安全性

- pause 中に pause するなどの不正操作を防ぐ
- セッション完了時の次状態を明確に定義する

### UI 品質

- デスクトップとモバイルの両方で崩れないこと
- 文字サイズ、余白、コントラストを UI モックに近づける
- ボタン状態を視覚的に分かりやすくする

## 結論

このプロジェクトでは、ブラウザ側 JavaScript でタイマーを制御し、Flask は画面配信とデータ管理に集中する構成が最も合理的である。

まずはシンプルな MVP を 1.pomodoro 配下で完成させ、その後に設定保存、統計、通知などを段階的に拡張する方針とする。
